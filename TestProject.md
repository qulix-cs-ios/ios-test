3. Нефункциональные требования
Функционал должен быть реализован стандарными средствами платформы. Использование сторонних библиотек НЕ допускается;

В приложении должен быть явно выделен интерфейс взаимодействия с сервером;

В приложении должна быть реализована стаб-реализация интерфейса сервера. Стаб имитирует работу сервера. Все данные хранит в оперативной памяти;

В стаб-реализации должны быть реализована задержка (1000мс), имитирующая работу сервера;

Первоначальные настройки хранятся в файле application.properties, в ресурсах приложения. Формат файла: стандартный java properties файл: свойство=значение (property=value);

Измененные настройки сохраняются в приватное хранилище приложения;

На последующих этапах, по усмотрению куратора, может быть добавлено требование реализации интерфейса, для взаимодействия с сервером по сети; При этом стаб и сетевая реализации должны существовать параллельно. Вероятно появление требования по переключению между реализациями.

В корне проекта должен быть создан файл readme.md. Содержимое файла:

краткая и точная инструкция по сборке

кодировка всех текстовых файлов: UTF-8 (без BOM)

5. Процесс разработки
5.1. Общие положения
Для выполнения тренировочного задания применяется итеративный подход

Длительность первой итерации: 1 неделя

Длительность каждой последующей итерации устанавливается куратором

По завершению каждой итерации должен быть выпущен билд (см. Выпуск билда)

В первой версии должно быть реализовано 100% функциональности приложения

Игнорирование требований не допускается

5.2. Итерация
Разработчик: получение задания или результатов обзора кода (code review)

Разработчик: пока не реализована вся функциональность или есть замечания: работа над проектом/замечаниями

Разработчик: в день выпуска билда, если реализованы все требования и исправлены все замечания, можно переходить к шагу 5, иначе к шагу 4

Разработчик: ежедневно, пока не выпущен билд, необходимо информировать куратора о причинах повлекших задержку выпуска билда, а так же о том что было сделано (по тренировочному проекту) за день.

Разработчик: Выпуск билда

Тестировщик: Проверка билда, регистрация дефектов и замечаний (issues) в системе отслеживания дефектов

Тестировщик: По завершению отправка отчета (критические дефекты, открытые, переоткрытые). Билд может быть отвергнут (rejected) при наличии критических дефектов. Если есть замечания уровнем выше Minor, переход к шагу 2

Куратор: обзор кода (code review).

Куратор: реализация удовлетворительна. Отправка уведомления о завершении проекта всем участникам. Завершение проекта.

Куратор: реализация НЕудовлетворительна. Регистрация замечаний в системе отслеживания дефектов;

Куратор: отсылка уведомления о том, что обзор кода завершен и назначение даты выпуска следующего билда. Переход к пункту 1.

5.3. Порядок работы над задачами
Открыть список задач (issues):

критерий: Opened AND assignee == Разработчик

сортировка: приоритет (самые важные в начале списка) Blocker, Critical, Major, Minor, Trivial

Выбрать первую задачу из списка;

Разобраться с сутью замечания, восполнить пробелы знаний в связанных темах;

Исправить проблему по всей системе. Например, если замечание "Магические числа в коде", необходимо исправить все подобные случаи;

Выполнить коммит изменений в VCS. В качестве комментария указать номер задачи и заголовок. Пример "#13 Магические числа в коде";

В комментарии к задаче описать, как была решена проблема;

Изменить поле assignee задачи на автора;

Перейти к п1.

5.4. Выпуск билда
Выпуск билда подразумевает следующие шаги:

проверка того, что:

все зарегистрированные замечания исправлены

проект собирается без ошибок

функциональность приложения работает правильно

определение номера билда (см. Нумерация билдов)

обновление файла version.txt

проверка того, что все изменения зафиксированы в VCS

фиксация версии в VCS (тэг с именем, соответствующим номеру билда)

email уведомление о выпуске билда

Формат письма:

Subject: Delivery notification. Build <build number>

Body:

Build <build number> available.
vcs: https://git.qulix.com/<login>/trainingtask.git
tag: <build number>
5.5. Нумерация билдов
номер билда представляет собой строку вида yyyymmdd-#, где:

yyyymmdd - текущая дата

- - знак минус

# - порядковый номер билда в рамках даты dd-mm-yyyy

номер билда глобальный для всего проекта

номер билда хранится в файле version.txt, который размещается в корне проекта

version.txt должен быть под контролем VCS (Version Control System в нашем случае Git)

формат файла: стандартный java properties файл: свойство=значение (property=value)

файл должен содержать свойство version, значением которого будет номер билда

файл может содержать комментарии

Пример содержимого version.txt:

version=20190312-1
 
6. Инфраструктура
 
6.1. Общие правила
исходный код проекта необходимо хранить в системе контроля версий Git (https://git.qulix.com)

логин/пароль - свои доменные, используемые для входа в систему

для тренировочного проекта необходимо создать проект (репозиторий) с именем 'trainingtask'

по специальному запросу необходимо предоставить доступ куратору проекта и тестировщику

замечания и дефекты регистрируются в Git репозитории проекта. Раздел Issues.

файлы добавляются в git репозиторий по задачам: по завершению реализации задачи, соответствующая группа файлов должна быть закоммичена в репозиторий

одна задача - один коммит

один коммит - одна задача

в любой момент времени проект должен собираться без ошибок компиляции

6.2. Структура проекта
Структура модуля должна соответствовать следующим правилам.

trainingtask/
    docs/                            >> проектная документация
    ${module-1}/                     >> проектный модуль 1
    ${module-2}/                     >> проектный модуль 2
    ...
    ${module-N}/                     >> проектный модуль N
        src/                         >> исходники
            main/                    >> основные исходники
                java/                >> java исходники
                resources/           >> ресурсы приложения (иконки, файлы интернационализации и пр.)
                webapp/              >> папка для веб приложения (jsp, html, web.xml и пр.)
            test/
                java/                >> исходники java (тесты)
                resources/           >> ресурсы для тестов
        lib/                         >> специальные библиотеки модуля
        build.gradle                 >> Gradle скрипт сборки проектного модуля
    .gitignore                       >> файл исключений (файлы|папки не хранящиеся в системе контроля версий)
    build.gradle                     >> Gradle скрипт сборки проекта
    settings.gradle                  >> Gradle скрипт настроек модулей
    readme.md                        >> файл описания проекта
    version.txt                      >> файл содержащий номер билда
